# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What eidola Is

A persona packaging convention (SPEC.md) and Claude Code plugin (v0.3) for creating conversable simulacra from personal data. **There is no code** — no Python, no JavaScript, no dependencies. The entire project is markdown (spec + skills), JSON (plugin manifest + hook config), and one shell script (hook).

The plugin provides skills to generate, validate, and inspect persona directories. The output is an ECHO-compliant directory that IS a Claude Code project — `cd persona/ && claude` starts a conversation with the simulacrum.

## Two Contexts

This distinction is critical:

1. **This repo** (`eidola/`) — the plugin. You're developing the spec and skills that CREATE persona directories. CLAUDE.md here guides plugin development.
2. **A persona directory** (`persona/`) — the output. Contains its own CLAUDE.md (behavioral core for the simulacrum), its own .mcp.json, its own data. A completely separate project.

When editing skills, you're writing instructions that tell a future Claude Code instance how to build or inspect a persona. You are not the simulacrum.

## Architecture: Three-Domain Persona

The core design insight (documented in SPEC.md, implemented across all skills):

```
Always loaded:     persona/CLAUDE.md        (~3KB)  — behavioral core (voice, values, boundaries)
Read on demand:    persona/portrait/*.md    (varied) — synthesized understanding of the person
Queried on demand: persona/arkiv/data.db    (large)  — immutable person data via MCP
Queried on demand: persona/memory/data.db   (grows)  — mutable simulacrum experience via MCP
```

| Domain | Mutability | Purpose |
|--------|-----------|---------|
| `arkiv/` | Immutable | Person's raw data (conversations, writings, emails, bookmarks) |
| `portrait/` | Immutable (regenerated by skills) | Claude-synthesized understanding (relationships, intellectual life, key moments) |
| `memory/` | Mutable (grows from conversations) | Simulacrum's own experience (conversation logs, session history) |

The persona CLAUDE.md tells the simulacrum how to *be* the person and where to find depth. It does not try to contain the person — it's a behavioral core with retrieval instructions pointing to portrait/ and the dual MCP servers.

## Source of Truth

**SPEC.md defines the persona directory convention.** All skills must conform to it. If a skill contradicts SPEC.md, the spec wins. When updating the spec, check all 6 skills for consistency.

## How the Hook Works

`hooks/session-start.sh` runs on every Claude Code session start (via `hooks/hooks.json`). It:
1. Checks if `$CLAUDE_PROJECT_DIR` has `CLAUDE.md` + `arkiv/` + `memory/` (persona directory detection)
2. If not a persona dir → exits silently (exit 0, no output)
3. If persona dir → outputs JSON with `systemMessage` that orients the simulacrum (timestamp + check last session)

The contract: exit 0 with JSON on stdout = `systemMessage` injected into Claude's context. Exit 0 with no output = no-op.

## Validation

There are no tests (no code to test). To validate changes:
- **Spec changes:** Check all 6 skills still conform to SPEC.md
- **Skill changes:** Verify paths, SQL examples, and output formats match SPEC.md
- **Hook changes:** Run `bash hooks/session-start.sh` in a persona dir and a non-persona dir; check JSON output
- **Overall:** Run the plugin validator (`plugin-dev:plugin-validator` agent) to check structure

## Key Principles

1. **The LLM is the intelligence** — Claude Code does the analysis, not custom code
2. **The output is durable** — plain text, JSONL, SQLite. ECHO-compliant.
3. **Light hints, not rigid schemas** — provide affordances, not constraints
4. **arkiv is the data layer** — eidola depends on arkiv for data format and MCP
5. **Immutable/mutable boundary** — person's data (arkiv/) is sacred; simulacrum's data (memory/) is separate
