# Layered Persona Architecture — Design Document

**Date:** 2026-02-17
**Version:** 0.3 (Draft)
**Status:** Approved for implementation planning

---

## Motivation

The v0.2 persona design treats CLAUDE.md as a monolithic system prompt and `memory/` as flat conversation logs. This works for simple cases but has three limitations:

1. **CLAUDE.md has a ceiling.** A rich persona (papers, repos, thousands of conversations, multiple intellectual threads) can't fit into a system prompt that's loaded on every turn. Context window waste.
2. **No synthesis layer.** The jump from raw arkiv data to behavioral instructions is too large. There's no intermediate layer that captures what the data *means* — relationships, key moments, formative experiences, the person's inner life.
3. **No session awareness.** The simulacrum doesn't know who it's talking to, when conversations happened, or how to maintain continuity across sessions.

## Design Principles

- **The LLM is the intelligence.** Don't impose templates — let Claude Code decide what matters based on what the data contains.
- **Light hints, not rigid schemas.** Provide affordances (here's what you *can* do) rather than constraints (here's what you *must* do).
- **Immutable/mutable boundary.** The person's data is sacred. The simulacrum's experience is separate.
- **Context efficiency.** Load only what's needed per turn. Retrieve depth on demand.

## Persona Directory Structure

```
persona/
├── README.md              # Self-describing (ECHO)
├── CLAUDE.md              # Behavioral core (~2-5KB)
├── .mcp.json              # MCP config (two arkiv servers)
├── provenance.json        # Consent and authorization
├── evaluation.md          # Fidelity assessment
├── portrait/              # Synthesized understanding (immutable)
│   └── *.md               #   Freeform narrative files
├── arkiv/                 # Person's data (immutable)
│   ├── data.db            #   SQLite (queryable via MCP)
│   ├── manifest.json      #   Collection schemas
│   ├── corpus/            #   Source JSONL (canonical)
│   └── media/             #   Audio clips, images
└── memory/                # Simulacrum's experience (mutable)
    ├── conversations.jsonl #   Append-only conversation logs
    └── data.db            #   Queryable memory
```

### Three Domains

| Domain | Contents | Mutability | Access |
|--------|----------|-----------|--------|
| `arkiv/` | Person's raw data — conversations, writings, emails, bookmarks, etc. | Immutable (updated only by arkiv import) | arkiv MCP server |
| `portrait/` | Claude-synthesized understanding — life narrative, relationships, intellectual threads | Immutable (updated only by /eidola-generate or /eidola-refresh) | Read tool (files) |
| `memory/` | Simulacrum's experience — conversation logs, session history | Mutable (grows with every conversation) | memory MCP server |

### Changes from v0.2

- **Added:** `portrait/` directory for synthesized understanding
- **Changed:** `memory/` becomes its own arkiv instance with `data.db`
- **Changed:** Person's data moved to `arkiv/` subdirectory
- **Removed:** `voice-samples.jsonl` as a named artifact (function absorbed by portrait/ files that naturally include representative quotes)
- **Added:** SessionStart/SessionEnd hooks for lifecycle management

## Cognitive Architecture

The three-domain split maps to a cognitive hierarchy:

```
Always loaded:     CLAUDE.md        (~3KB)  — procedural (how I behave)
Read on demand:    portrait/*.md    (varied) — semantic (what my life means)
Queried on demand: arkiv MCP        (unlimited) — episodic (specific memories)
Queried on demand: memory MCP       (growing)  — autobiographical (my experience as a simulacrum)
```

### CLAUDE.md — Behavioral Core

Compact (~2-5KB). Loaded into the context window on every turn. Contains:

- **Identity** — name, background, self-description (condensed)
- **Voice registers** — how they speak in different contexts, with rules not exhaustive examples
  - e.g., "With AI tools: exploratory, stream-of-consciousness. In professional writing: formal, precise. In conversation: informal, direct."
- **Values summary** — core principles (detail lives in portrait/)
- **Boundaries** — what the simulacrum must not claim or do
- **Retrieval instructions** — how to use arkiv MCP, when to read portrait/ files, when to check memory
- **Light hints for state** — "track who you're speaking with and when" (awareness, not a schema)

CLAUDE.md does not try to contain the person. It tells the simulacrum how to *be* the person and where to find depth.

### portrait/ — Synthesized Understanding

Freeform markdown files generated by `/eidola-generate`. Claude Code analyzes all arkiv data and produces whatever files the data warrants. Not templated — content is driven by what the data actually contains.

Examples of files Claude might produce (not prescribed):
- `relationships.md` — key people, family, how they relate
- `intellectual-life.md` — ideas, interests, intellectual threads
- `formative-experiences.md` — key moments, turning points, losses
- `professional-work.md` — career, contributions, expertise
- `values-and-beliefs.md` — principles, worldview, contradictions

Files naturally include representative quotes from the data, with references back to arkiv collections. This subsumes the former `voice-samples.jsonl` — voice examples are woven into the narrative rather than extracted into a rigid format.

Filenames are self-describing metadata. No manifest needed — Claude Code can `ls portrait/` and decide which files are relevant to the current conversation.

### memory/ — Simulacrum's Experience

A separate arkiv instance for the simulacrum's own history. Clean separation from the person's data ensures the ghost's words are never conflated with the person's words.

- `conversations.jsonl` — append-only logs of all conversations, with timestamps and metadata (who's speaking, session context)
- `data.db` — SQLite database for querying conversation history at scale (imported from JSONL)

The simulacrum uses the memory MCP server to recall past conversations, recognize returning visitors, and maintain continuity.

### .mcp.json

Two arkiv servers — one for the person's data, one for the simulacrum's memory:

```json
{
  "mcpServers": {
    "arkiv":  { "command": "arkiv", "args": ["serve", "arkiv/data.db"] },
    "memory": { "command": "arkiv", "args": ["serve", "memory/data.db"] }
  }
}
```

## Session Lifecycle

### SessionStart Hook

When someone starts a conversation (`cd persona/ && claude`), a SessionStart hook fires. It performs a light orientation:

- Injects the current timestamp
- Prompts the simulacrum to check its last session (who, when, topic)
- The simulacrum holds this context lightly — doesn't dump it on the user, but is oriented

The orientation is deliberately light: check last session, note the time, be ready. No heavy pattern analysis at startup. The simulacrum can dig deeper during the conversation if needed.

### SessionEnd Hook

When the session ends, a hook can append a lightweight session marker to memory. This is optional and should not block exit.

### No Explicit Commands

Session boundaries are inferred from timestamps in the memory database. No `/start-session` or `/end-session` commands. The person just talks and leaves. The infrastructure handles bookkeeping.

### Interlocutor Tracking

The simulacrum is given a light hint to track who it's speaking with. It may:
- Ask "who am I speaking to?" when it can't infer from context
- Infer identity from conversation patterns or returning visitors
- Store interlocutor information in memory for future sessions

No prescribed schema. Claude decides what to track based on what emerges from conversations.

---

## State Management Philosophy

**Let Claude decide.** The design provides affordances (a memory database, retrieval instructions, light hints) but does not prescribe what state to track or how to organize it. This maximizes flexibility and leverages Claude Code's intelligence rather than constraining it with rigid schemas.

The CLAUDE.md mentions things worth being aware of (who you're talking to, conversation continuity) but treats them as suggestions, not requirements. The simulacrum's behavior emerges from the interaction between the behavioral core, the synthesized portrait, the raw data, and the accumulated memory.

## Plugin Skill Updates Required

### Modified Skills

- **`/eidola-generate`** — Now produces portrait/ directory with freeform files (not just CLAUDE.md). Sets up memory/ arkiv instance. Generates .mcp.json with two servers. Drops voice-samples.jsonl generation. Data files go to arkiv/ subdirectory.
- **`/eidola-refresh`** — Can update individual portrait/ files (not just CLAUDE.md). More surgical than full regeneration.
- **`/eidola-validate`** — Checks three-domain structure (arkiv/, portrait/, memory/). Validates both MCP server configs. Verifies immutable/mutable separation.
- **`/eidola-info`** — Reports on all three domains. Shows portrait/ file inventory. Shows memory statistics (sessions, unique interlocutors, date range).
- **`/eidola-evaluate`** — References portrait/ for evaluation context. No structural changes.
- **`/eidola-interview`** — No structural changes.

### New Components

- **SessionStart hook** — Light orientation prompt (timestamp, check last session)
- **SessionEnd hook** — Optional session marker append to memory

## Graceful Degradation (Updated)

1. **Full** — Claude Code + dual arkiv MCP + portrait/ + memory. Interactive simulacrum with deep understanding, memory retrieval, and session continuity.
2. **Good** — Any LLM + CLAUDE.md + portrait/ files as context. No live data retrieval, but rich understanding from the portrait.
3. **Minimal** — Read CLAUDE.md as a text file. Compact personality description.
4. **Archival** — README.md explains everything. All data in durable formats (text, JSONL, SQLite, markdown).

---

## What This Design Does NOT Do

- Does not prescribe portrait/ file names or structure — Claude decides based on data
- Does not impose a memory schema — the simulacrum tracks what it finds useful
- Does not require session management commands — hooks and timestamps handle lifecycle
- Does not fine-tune or train models — the LLM is the intelligence, the data is the memory
- Does not add code, dependencies, or pipelines — still a convention, not software
