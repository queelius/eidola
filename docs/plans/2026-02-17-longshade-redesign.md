# Design: longshade Redesign

**Date:** 2026-02-17
**Status:** Approved

---

## Summary

longshade is a persona packaging convention and Claude Code plugin for creating simulacra from personal data.

- **The spec** defines what a persona directory looks like
- **The plugin** is the reference implementation (Claude Code skills)
- **The output** is an ECHO-compliant, self-contained directory that IS a Claude Code project

To use a persona: `cd persona/ && claude`

---

## Motivation

The previous longshade was over-engineered: custom RAG (FAISS, embeddings, chunker), 7 specialized ingesters, approach-specific output directories. All of that is now unnecessary because:

1. **arkiv** handles the data layer (JSONL canonical, SQLite queries, MCP server)
2. **Claude Code** is the intelligence layer (retrieval via MCP, generation via conversation)
3. **The LLM gets better every year** вҖ” any custom pipeline we build today will be obsolete tomorrow

longshade's job is not to be smart. It's to package data clearly so a smart LLM can use it.

---

## What longshade Is

A persona packaging convention (SPEC.md) and Claude Code plugin (reference implementation).

**A persona directory is a Claude Code project.** When you `cd persona/ && claude`, Claude loads the system prompt from CLAUDE.md, connects to the person's data via arkiv MCP, and becomes the simulacrum.

## What longshade Is NOT

- Not a Python package (no pip install, no dependencies)
- Not an application that calls LLM APIs
- Not a RAG pipeline, embedding system, or retrieval engine
- Not a fine-tuned model or training framework
- Not specific to any LLM provider (the output is portable)

```
в”ңв”Җв”Җ .mcp.json              # Connects arkiv MCP server to data.db
в”ңв”Җв”Җ manifest.json          # arkiv manifest (collection schemas)
в”ңв”Җв”Җ data.db                # arkiv SQLite (queryable via MCP)
в”ңв”Җв”Җ voice-samples.jsonl    # Few-shot Q&A pairs demonstrating voice
в”ңв”Җв”Җ memory/                # Online memory (conversations with the simulacrum)

```

### Required files

| File | Purpose |
|------|---------|
| `README.md` | Self-describing. What this persona is, how to use it. ECHO-compliant. |
| `CLAUDE.md` | System prompt. Personality, voice characteristics, values, boundaries. |
| `data.db` | arkiv SQLite database. The person's data, queryable. |
| `manifest.json` | arkiv manifest. Collection descriptions and schemas. |

### Optional files

| File | Purpose |
|------|---------|
| `.mcp.json` | MCP server config. Points arkiv at data.db. |
| `voice-samples.jsonl` | Few-shot Q&A pairs demonstrating voice and tone. |
| `media/` | Audio clips, images, referenced by arkiv records. |
| `corpus/` | Source JSONL files. Canonical representation of the data. |

### .mcp.json

```json
{
  "mcpServers": {
    "arkiv": {
      "command": "arkiv",
      "args": ["serve", "data.db"]
    }
  }
}
```

### CLAUDE.md

The system prompt is the heart of the persona. It should include:

- Identity (name, description, background)
- Voice characteristics (communication style, vocabulary, tone)
- Values and beliefs (with examples from the data)
- Intellectual interests and expertise areas
- Characteristic phrases and patterns
- Boundaries (what the echo should not claim or do)
- Instructions for using arkiv MCP tools to retrieve memories

This is generated by an LLM analyzing the person's arkiv data. The LLM does the hard work of distillation; longshade provides the skill that guides the process.

### voice-samples.jsonl

```jsonl
{"question": "What do you think about AI consciousness?", "answer": "I'm skeptical of strong claims...", "source": "conversation-456"}
{"question": "Why do you care about archiving?", "answer": "The things we create should outlast us...", "source": "essay-789"}
```

Few-shot examples that demonstrate the persona's voice. Used for in-context learning.

### Online Memory

When someone talks to the simulacrum, those conversations become new data вҖ” stored in `memory/` as arkiv-format records. Key properties:

- **Marked as generated.** `metadata.generated: true` distinguishes simulacrum output from original person data.
- **Stored as arkiv records.** Same universal format. Can be imported into data.db for retrieval.
- **Provides continuity.** The simulacrum remembers past conversations via MCP queries.
- **Grows over time.** The persona accumulates new "memories" from interactions.

Online memory is optional. Without it, each conversation starts fresh from the original data.

## Graceful Degradation

The persona degrades gracefully across four levels:

1. **Full experience** вҖ” Claude Code + arkiv MCP. Interactive simulacrum with memory retrieval.
2. **Good experience** вҖ” Any LLM with CLAUDE.md as system prompt + voice-samples.jsonl as few-shot.
3. **Minimal experience** вҖ” Read CLAUDE.md as a text file. It's a plain-text personality description.
4. **Archival** вҖ” README.md explains everything. JSONL and SQLite are human-readable and durable.

---

## Voice Synthesis Hooks

The spec acknowledges that voice synthesis is possible but does not prescribe an approach.

If `media/` contains audio reference clips of the person's voice, these can be used with any TTS system (XTTS, RVC, F5-TTS, ElevenLabs, etc.) for speech synthesis. A TTS MCP server could be added to `.mcp.json` alongside arkiv to give the simulacrum a voice.

arkiv provides the data. The TTS integration is left to the user and future tooling.

## Claude Code Plugin

The longshade repo is a Claude Code plugin with three skills:

### /longshade-generate

Create a persona from arkiv data. Interactive, conversational process:

1. Point at arkiv data (manifest.json, JSONL files, or SQLite DB)
2. Claude analyzes the data вҖ” reads samples, identifies patterns, distills personality
3. Claude writes CLAUDE.md (system prompt) with user review and refinement
4. Claude curates voice-samples.jsonl from representative Q&A pairs
5. Copies/links media files
6. Writes manifest.json, .mcp.json, README.md
7. Output: a complete persona directory

The user stays in control throughout. This is a conversation, not a pipeline.

### /longshade-validate

Check a persona directory against the spec:

- Required files present (README.md, CLAUDE.md, data.db, manifest.json)
- CLAUDE.md is well-formed (has identity, voice, values, boundaries sections)
- data.db is a valid arkiv SQLite database
- manifest.json matches the data
- .mcp.json points to valid paths

### /longshade-info

Inspect a persona directory:

- Data stats (record count by collection, date range, mime types)
- Voice sample count
- Media inventory (audio clips, images)
- Collection summaries from manifest

## Dependencies

- **arkiv** вҖ” data layer (JSONL, SQLite, MCP server). Must be installed for `arkiv serve`.
- **Claude Code** вҖ” intelligence layer. The plugin runs inside Claude Code.

No other dependencies.

---

## ECHO Compliance

The persona directory is ECHO-compliant by construction:

- **Self-describing**: README.md explains what it is and how to use it
- **Durable formats**: plain text (CLAUDE.md), JSONL, SQLite
- **Graceful degradation**: works at four levels, from full MCP to just reading text files
- **Local-first**: everything is local files, no cloud dependency

## Open Questions

1. **CLAUDE.md size limits.** How much personality data can fit in CLAUDE.md before hitting context limits? May need to reference external files or rely more on MCP retrieval.

2. **Offline persona.** If arkiv MCP isn't running, the persona still works via CLAUDE.md alone вҖ” but with no memory retrieval. Should the plugin auto-start arkiv serve?

3. **Multi-person.** Could a persona directory contain multiple people? Probably not вҖ” one directory per person, compose at a higher level.

4. **Updates.** When new data arrives (new conversations, new writings), how to update the persona? Re-run /longshade-generate? Incremental update?
