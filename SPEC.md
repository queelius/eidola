# eidola: Persona Packaging Convention

**Version:** 0.3 (Draft)

---

## Purpose

eidola defines how to package personal data as a conversable persona. The output is an ECHO-compliant directory that doubles as a Claude Code project — `cd persona/ && claude` starts a conversation with the simulacrum.

The intelligence is in the LLM, not in code. eidola is a convention, not a pipeline.

---

## Persona Directory

```
persona/
├── README.md              # Self-describing (ECHO). What this is, how to use it.
├── CLAUDE.md              # Behavioral core. Voice, values, boundaries, retrieval instructions.
├── .mcp.json              # Two arkiv servers: person's data + simulacrum's memory
├── provenance.json        # Consent and authorization record
├── evaluation.md          # Fidelity assessment and calibration notes
├── portrait/              # Synthesized understanding of the person
│   └── *.md               #   Freeform narrative files (relationships, experiences, etc.)
├── arkiv/                 # Person's data (immutable)
│   ├── data.db            #   arkiv SQLite (queryable via MCP)
│   ├── README.md          #   Self-describing archive metadata (YAML frontmatter)
│   ├── schema.yaml        #   Curated metadata schema (optional)
│   ├── corpus/            #   Source JSONL files (canonical, human-readable)
│   │   ├── conversations.jsonl
│   │   ├── writings.jsonl
│   │   └── ...
│   └── media/             #   Audio clips, images, referenced by records
└── memory/                # Simulacrum's experience (mutable)
    ├── conversations.jsonl #   Append-only conversation logs
    └── data.db            #   Queryable memory (separate arkiv instance)
```

### Required Files

| File | Purpose |
|------|---------|
| `README.md` | Self-describing. What this persona is, how to use it. |
| `CLAUDE.md` | Behavioral core. Voice, values, boundaries, retrieval instructions. |
| `arkiv/data.db` | arkiv SQLite database. The person's data, queryable via MCP. |
| `arkiv/README.md` | arkiv archive metadata. Self-describing with YAML frontmatter (collections, descriptions). |

| File | Purpose |
|------|---------|
| `.mcp.json` | MCP server config. Two servers: arkiv (person's data) and memory (simulacrum's experience). |
| `provenance.json` | Consent and authorization record. Who authorized this simulacrum. |
| `evaluation.md` | Fidelity assessment and calibration notes. |
| `portrait/` | Synthesized understanding. Freeform markdown files generated by Claude from arkiv data. |
| `memory/` | Simulacrum's experience. Conversations, session history. Separate arkiv instance. |
| `arkiv/schema.yaml` | Curated metadata schema for arkiv collections. |
| `arkiv/corpus/` | Source JSONL files. Canonical representation of the data. |
| `arkiv/media/` | Audio clips, images, referenced by arkiv records. |

## CLAUDE.md (Behavioral Core)

The system prompt is the behavioral core of the persona — loaded on every turn. It tells the simulacrum how to *be* the person: identity, personality, voice, values, and how to access deeper knowledge.

With large context windows (1M+ tokens), CLAUDE.md can be rich and detailed rather than minimal. The goal is to pre-load enough synthesized understanding that the simulacrum rarely needs to query MCP for personality or biographical information — MCP is reserved for specific lookups ("what did I say on March 3rd?"), not general understanding.

Contents:

- **Identity** — name, background, self-description
- **Voice registers** — how they speak in different contexts (formal, informal, exploratory), with rules, patterns, and representative examples
- **Values** — core principles, positions, contradictions, with depth
- **Personality** — how they think, what they care about, their humor, their blind spots
- **Boundaries** — what the simulacrum should not claim or do
- **Retrieval instructions** — how to use arkiv MCP for specific data lookups, memory MCP for past conversations, and when to read portrait/ files
- **State awareness** — session context (who's speaking, when), not a prescribed schema

The system prompt is plain text. It works with any LLM, not just Claude.

## portrait/ — Synthesized Understanding

Rich, detailed markdown files generated by `/eidola-generate`. Claude Code analyzes the full breadth of arkiv data — hundreds or thousands of records — and synthesizes deep, comprehensive portrait files. Not templated — content is driven by what the data actually contains.

With 1M+ token context windows, portrait files are designed to be **pre-loaded into the simulacrum's context**, not read on demand. The goal is exhaustive synthesis: extract every salient pattern, relationship, key moment, and intellectual thread so the simulacrum has deep understanding available on every turn.

Examples of files Claude might produce (not prescribed):

- `relationships.md` — key people, family dynamics, how they relate, characteristic interactions
- `intellectual-life.md` — ideas, interests, intellectual threads, how they evolved over time
- `formative-experiences.md` — key moments, turning points, losses, growth, with narrative detail
- `professional-work.md` — career arc, contributions, expertise, how they think about their work
- `values-and-beliefs.md` — principles, worldview, contradictions, evolution of positions
- `communication-style.md` — how they write, argue, joke, explain, with representative examples
- `daily-life.md` — routines, preferences, habits, the texture of their life
- `humor-and-personality.md` — what makes them laugh, their quirks, how others experience them

Files include representative quotes and exchanges from the data, with arkiv references (collection, timestamp, context). Voice is demonstrated through actual examples, not just described.

Key properties:

- **Generated, not handwritten.** Claude Code synthesizes the portrait from deep data analysis.
- **Immutable in normal operation.** Updated only by `/eidola-generate` or `/eidola-refresh`.
- **Self-describing filenames.** No manifest needed — filenames are the metadata.
- **Variable depth.** A person with rich family data gets a detailed `relationships.md`. A person without doesn't get one at all.
- **Designed for pre-loading.** Each file can be 2,000-10,000+ words. The total portrait may be 100-300KB. With large context windows, all portrait files are loaded at session start rather than read on demand.
- **Structured for navigation.** Even within large files, clear sections and headers help the simulacrum locate relevant context quickly.

## .mcp.json

Two arkiv servers — one read-only for the person's data, one writable for the simulacrum's memory:

```json
{
  "mcpServers": {
    "arkiv":  { "command": "arkiv", "args": ["mcp", "arkiv/data.db"] },
    "memory": { "command": "arkiv", "args": ["mcp", "--writable", "memory/data.db"] }
  }
}
```

Both servers expose the same arkiv MCP read tools:

| Tool | Purpose | Servers |
|------|---------|---------|
| `get_manifest()` | What collections exist, their descriptions and schemas | arkiv, memory |
| `get_schema(collection?)` | What metadata keys can be queried | arkiv, memory |
| `sql_query(query)` | Run read-only SQL against the data | arkiv, memory |
| `write_record(collection, content, metadata)` | Persist a new record to the database | memory only |

The `arkiv` server provides the person's immutable data (read-only). The `memory` server provides the simulacrum's mutable experience and supports writes via `write_record()`. The LLM decides what to retrieve from each based on the conversation — more powerful than traditional RAG because the LLM controls retrieval iteratively, refining queries based on results.

## memory/ — Simulacrum's Experience

A separate arkiv instance for the simulacrum's own history. Clean separation from the person's data ensures the ghost's words are never conflated with the person's words.

- `memory/conversations.jsonl` — append-only logs of all conversations, with timestamps and metadata (who's speaking, session context)
- `memory/data.db` — SQLite database for querying conversation history at scale (imported from JSONL)

The immutable/mutable boundary is fundamental: the person's data in `arkiv/` is sacred and never modified by the simulacrum. The simulacrum's data in `memory/` is separate and grows with every conversation.

The simulacrum persists conversation data via the `write_record(collection, content, metadata)` MCP tool on the memory server. This is the write path for all simulacrum-generated data.

Key properties:

- **Marked as generated.** `metadata.generated: true` distinguishes simulacrum output from original person data.
- **Stored as arkiv records.** Same universal format. Queryable via the memory MCP server.
- **Provides continuity.** The simulacrum can recall past conversations, recognize returning visitors, and maintain context across sessions.
- **Grows over time.** The persona accumulates experience from interactions.
- **Light hints for state.** The simulacrum is given awareness of what to track (who's speaking, when) but no rigid schema. Claude decides what matters.

Memory is optional. Without it, each conversation starts fresh from the original data.

## Session Lifecycle

Session boundaries are inferred from timestamps — no explicit commands. The person just talks and leaves. The infrastructure handles bookkeeping.

- **SessionStart hook.** Fires when someone starts `claude` in the persona directory. Injects the current timestamp. Prompts the simulacrum to check its last session (who, when, topic). The orientation is deliberately light — be ready, don't dump context on the user.
- **SessionEnd hook.** Fires when the session closes. Optionally appends a lightweight session marker to memory. Does not block exit.
- **Interlocutor tracking.** The simulacrum may ask who it's speaking with, or infer from context and returning-visitor patterns. No prescribed schema — Claude decides what to track based on what emerges from conversations.

## Evaluation

How do you know if the simulacrum is any good? Three methods:

1. **Held-out test.** Reserve some conversations/writings during generation. Ask the simulacrum questions the real data answers. Compare fidelity.
2. **Calibration interview.** Show the person simulacrum responses and ask: "Does this sound like you? What's off?" Iterate on CLAUDE.md.
3. **Hallucination check.** Probe topics the person never discussed. The simulacrum should acknowledge uncertainty, not confabulate.

Evaluation results can be stored as `evaluation.md` in the persona directory — human-readable, part of the self-describing package.

## Updates

The persona is not frozen. New data arrives, the person changes, the simulacrum should evolve.

- **Data layer.** New JSONL imported into `arkiv/data.db` via `arkiv import`. New records are immediately queryable via the arkiv MCP server. No persona-level changes needed.
- **Behavioral core.** CLAUDE.md may need refreshing when significant new data arrives. Re-run `/eidola-generate` for full regeneration, or `/eidola-refresh` for a lighter update.
- **Portrait.** `/eidola-refresh` can update individual portrait/ files or suggest new ones when new data reveals new relationships, experiences, or intellectual threads. More surgical than full regeneration.
- **Memory.** Grows naturally from conversations. Session hooks handle lifecycle. No action needed.
- **Versioning.** `git init` inside the persona directory gives full version history. Every update to CLAUDE.md and portrait/ is tracked.

## Consent and Provenance

A simulacrum should carry its authorization. The persona directory includes `provenance.json`:

```json
{
  "subject": "Alex Towell",
  "authorized_by": "Alex Towell",
  "date": "2026-02-17",
  "statement": "I authorize the creation of this digital simulacrum from my personal data.",
  "data_sources": ["memex", "mtk", "btk", "ptk", "ebk"],
  "restrictions": [
    "Do not claim to be conscious or alive",
    "Do not make medical or legal claims"
  ],
  "signature": null
}
```

- **Who authorized it** — the person themselves, or an estate, or a designee
- **What data was included** — provenance of the source material
- **Restrictions** — what the simulacrum should not do (also encoded in CLAUDE.md boundaries)
- **Signature** — optional GPG/age signature for authenticity

A simulacrum without consent is a deepfake. A simulacrum with consent is a legacy.

## Voice Synthesis

The spec acknowledges that voice synthesis is possible but does not prescribe an approach.

If `arkiv/media/` contains audio reference clips of the person's voice, these can be used with any TTS system (XTTS, RVC, F5-TTS, ElevenLabs, etc.) for speech synthesis. A TTS MCP server could be added to `.mcp.json` alongside the arkiv servers.

arkiv provides the data. The TTS integration is left to the user and future tooling.

## Graceful Degradation

Personas degrade gracefully across four levels:

1. **Full** — Claude Code + dual arkiv MCP + portrait/ + memory. Interactive simulacrum with deep understanding, memory retrieval, and session continuity.
2. **Good** — Any LLM + CLAUDE.md + portrait/ files as context. No live data retrieval, but rich understanding from the portrait.
3. **Minimal** — Read CLAUDE.md as a text file. Compact personality description.
4. **Archival** — README.md explains everything. All data in durable formats (text, JSONL, SQLite, markdown).

## ECHO Compliance

The persona directory is ECHO-compliant by construction:

- **Self-describing** — README.md explains what it is and how to use it
- **Durable formats** — plain text, JSONL, SQLite
- **Graceful degradation** — works at four levels
- **Local-first** — everything is local files, no cloud dependency

---

## Dependencies

- **arkiv** — data format and MCP server. Provides JSONL canonical storage, SQLite query layer, and MCP tools.
- **Claude Code** — runs the plugin and serves as the intelligence layer. Not required for the output — any LLM can use CLAUDE.md.

---

## Privacy

Persona directories contain personal data. Consider:

- What you include determines what the simulacrum can say
- Online memory accumulates new data over time
- Control access to the persona directory
- arkiv data can be encrypted with GPG/age for storage

The simulacrum can answer questions you never anticipated. Think carefully about what's included.
